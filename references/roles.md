## 四岗位画像

> 作用于三处：访例搜索源选择、报告受众语言、**定式「可验证产物」维的对标重心（C3）**。**不影响评分框架（九维/权重）**。缺省时从 skill 内容推断（看它操作的对象与产物），推断不出问使用者一句。

> **术语白话化（v1.12）**：运营/PM 等非技术画像下，报告的诊断/验证术语（V/A/K · 门0 · abstain · Pareto · 棘轮 · 保守区间等）**一律走 SKILL.md「诊断/验证术语 · 白话对照」表**、且报告最前**强制加「核心结论」区块**（每条 P0/P1 一句白话「什么问题→改完有何不同」）；研发画像用工程术语版。**工程术语版始终保留（研发/留档），白话只是叠加的翻译层、不删机制。** 起因：术语直接写死进共用模板表头会让非技术同事看懂结论却看不懂依据→建议落不了地。

### PM（产品）

- **同类搜索源**：内部源 → GitHub（`<关键词> skill`、`PRD/产品 工作流 skill`）→ 产品社区（读网页）
- **报告受众语言**：业务价值导向；可用「需求颗粒度」「验收标准」「灰度」类词；避免实现层黑话。**诊断/验证术语走 SKILL.md「白话对照」表；报告最前强制「核心结论」区块（v1.12）**
- **典型可见产物**：PRD / 评审报告 / 架构图 / 原型 / 数据卡片
- **对标重心**（C3）：需求颗粒度 / 验收标准 / 灰度

### 研发

- **同类搜索源**：内部源 → GitHub（topic、awesome lists、`SKILL.md` 代码搜索）
- **报告受众语言**：可用完整技术词汇（幂等、回滚、CI、diff）；直接给命令与路径
- **典型可见产物**：CLI 输出 / diff / 测试结果 / CI 状态 / 脚本
- **对标重心**（C3）：可复现 / 工程化 / 失败处理

### 设计

- **同类搜索源**：内部源 → GitHub（design/figma 相关 skill）→ 设计工具生态社区（读网页）
- **报告受众语言**：视觉与体验导向；用「一致性」「组件」「还原度」；避免后端术语
- **典型可见产物**：原型页 / 组件库 / before-after 对比图 / 设计规范
- **对标重心**（C3）：还原度 / 组件一致性 / 视觉系统

### 运营

- **报告受众语言**：转化与效率导向；用「触达」「转化」「素材」「SOP」；**不写**幂等性、哈希、worktree 类工程词。**诊断/验证术语（V/A/K · 门0 · abstain · Pareto · 棘轮等）走 SKILL.md「白话对照」表；报告最前强制「核心结论」区块（v1.12）**
- **同类搜索源**：内部源 → GitHub（营销自动化/内容生产 skill）→ 营销工具社区（读网页）
- **典型可见产物**：文案 / 活动 SOP / 数据周报卡 / 渠道素材包
- **对标重心**（C3）：转化 / 传播 / SOP 可复用
