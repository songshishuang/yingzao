# README 出门标准（落成第 8 节与开源出门清单的依据文件）

> 营造给被打磨 skill 提 README 建议时的统一标准。**全部条款经实战蒸馏**：营造自身 README 四轮迭代（痛点开场→四样东西→引擎章→数据钩子）+ 多场大修的 README 建议实践 + 访例中高星项目的共性观察。不照搬任何单一项目的清单——标准只收"在我们手里验证有效"的。

## 一、首屏 10 秒法则（开源目标的验证门②）

读者扫 10 秒就要拿到三件事：**这是干嘛的（一句话）→ 凭什么信（数字/产物）→ 怎么开始（一条命令）**。

| 要件 | 标准 | 反例 |
|------|------|------|
| 一句话定位 | 讲"替用户省掉什么痛苦"，不罗列功能 | "一个功能强大的多用途工具" |
| 数字钩子 | 真实可复核的数据（分数前后对比/实测比值），紧跟定位句 | 形容词堆砌（"大幅提升""显著优化"） |
| 一条命令 | 首屏可见的安装命令块 | 安装方法埋在第 8 节 |

## 二、结构骨架（按读者旅程排，不按开发者心智排）

1. **痛点场景**（2-4 条，读者对号入座；覆盖单人与团队两类视角）
2. **它给你什么**（交付物表——"动完手的成品"要显性，不能藏在过程描述里）
3. **快速开始**（装 + 第一句话怎么说）
4. **工作流程**（每个环节三段式：干什么 / 对你的价值 / 产出物——只讲机制名词不讲价值的环节读者会跳过）
5. **真实战绩**（数据表 + 1-2 个可复述的故事；故事比表格传播得远）
6. **安全承诺**（"永远不会做什么"清单——信任来自边界而非能力）
7. **与同类的区别**（对比表，讲差异不贬低）
8. References（方法论有外部来源时列出处——社区惯例与信誉管理，非义务）

## 三、证据纪律（营造特色，必查）

- 所有数字必须可溯源（指向台账/报告/运行记录）；**写不出处的数字删掉**
- demo/截图必须真实运行产物，录制脚本入库保证可复现——**摆拍一经发现按 P0 处理**
- 机制图（流程/循环）优先自绘 SVG：信息层次可控、缩放无损、可入库迭代
- "进行中"的事实（计数器/进度）写明当前值，让承诺变成可验证的进行时

## 四、跨 runtime 中立（预检 P5 同源）

- 不写"仅支持 X 工具"类锁定措辞（触发词区豁免）；多平台路径并列是正当的
- 安装段覆盖主流 runtime 或给 `--dest` 类逃生门

## 五、内部 skill 的轻量版（internal 目标）

内部 skill 不强制 README——验证门②查"触发方式 + 使用边界 + 最小示例"三件 10 秒可懂，载体可以是 SKILL.md 头部。走向开源时再按本标准补全（出门清单逐项给差距）。
