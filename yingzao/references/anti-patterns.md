# 禁例 · 红线清单

> 本文件分两区：**出厂区**（v1.0 固化，修改走版本变更）与**增量区**（自迭代沉淀——只能由大修收尾自检生成候选 diff、经使用者确认后写入；查勘收尾零写入）。

## 出厂区

1. **不要只改 SKILL.md 不看 README 与展示物**——skill 是整体资产，文档与产物一起评。
2. **不要只做静态检查不跑测试 prompt 就给高分**——无实测总分上限 70，估分必须显式标注。
3. **不要只找一个同类就下定位结论**——访例至少覆盖内部源与公网两路；搜不到如实标「未访得」。
4. **不要把「功能更多」当「更好」**——验证门只认「输出优于原版」，不认功能堆叠。
5. **不要把私有路径、账号、密钥写进公开 skill**——预检密钥扫描 FAIL 即停；外发报告一律脱敏。
6. **不要在 README 写「全自动解决一切」类不可信大词**——可验证的具体声明才进首屏。
7. **不要一轮细作改多个变量**——单变量纪律是归因的前提；批量授权也只是连续应用，不是合并改写。
8. **不要让改写者与验收者是同一视角**——盲评分离；自打磨时强制陌生用户视角。
9. **不要照搬任何同类 skill 的名字、叙事与结构**——学手艺不偷皮；对标只取机制思想。
10. **不要把使用者的疑问句当成授权**——祈使句才放行；一次授权只覆盖声明范围。
11. **不要新增「三处不可见」的暗开关**——任何可选能力必须同时出现在 README 触发清单 / 运行时播报 / 报告固定位。
12. **营造运行时永不修改自身 .gitignore**——该文件出厂预置；行为规则变更只走候选 diff。
13. **不要在隔离实例外冒充裸基线**——打磨 Agent 读过目标全文，「假装不知道」做不到；无隔离能力一律估分。
14. **不要让测试运行执行真实副作用**——安全闸默认 dry-run；发布/删除/外呼类动作必须单独祈使授权。
15. **不要在回答里塞大修产物硬撑**——工作区全不可写就降档为查勘+方案报告，patch 与 manifest 不进对话流。
16. **落架不破棘轮**（v1.1）——重写版必须同套测试同台对比、严格优于现版才采用；「重写的更顺手 / 结构更漂亮」不是采用理由，弃稿恢复现版不算失败。
